# SPI – Serial Peripheral Interface (Conceptual Deep Dive)

## Serial Peripheral Interface

## Core Idea

SPI is a **synchronous, full-duplex, master-slave communication protocol** designed for **high-speed short-distance communication** between microcontrollers and peripherals.

---

## Physical Lines (4-wire system)

| Line    | Direction      | Purpose               |
| ------- | -------------- | --------------------- |
| MOSI    | Master → Slave | Data sent to slave    |
| MISO    | Slave → Master | Data returned         |
| SCK     | Master → Slave | Clock signal          |
| SS / CS | Master → Slave | Select specific slave |

Clock is generated by the **master only**.

---

## Why "Synchronous"?

Because communication is coordinated by a **shared clock (SCK)**.

Each clock pulse:

- Shifts out 1 bit
- Shifts in 1 bit

This makes SPI:

- Very fast
- Deterministic
- Timing-stable

---

## Data Transfer Mechanism (Shift Registers)

Internally:

- Master and slave each have a shift register.
- On every clock edge:
  - Master shifts 1 bit out (MOSI)
  - Slave shifts 1 bit out (MISO)
  - Both shift registers update simultaneously

This is why SPI is **full duplex**.

---

## SPI Modes (Clock Polarity & Phase)

Two parameters define timing:

- CPOL → Clock polarity
- CPHA → Clock phase

These define **4 SPI modes (0–3)**.

This matters because:

- Some devices sample on rising edge
- Some sample on falling edge

Incorrect mode = corrupted data.

---

## Multi-Slave Architecture

SPI supports multiple slaves by:

- Sharing MOSI, MISO, SCK
- Giving each slave a separate **CS (Chip Select)**

Master selects one slave at a time.

---

## Strengths

✔ Very high speed

✔ Simple hardware

✔ No addressing overhead

✔ Deterministic timing

---

## Weaknesses

✖ Requires more wires

✖ No built-in error checking

✖ No standard protocol layer (device-specific commands)

---

## Typical Use Cases

- SD cards
- TFT displays
- ADC/DAC modules
- Flash memory

---

# I2C – Inter-Integrated Circuit (Conceptual Deep Dive)

## Inter-Integrated Circuit

## Core Idea

I2C is a **synchronous, half-duplex, multi-master, multi-slave communication bus** designed for **efficient wiring and device addressing**.

---

## Physical Lines (2-wire system)

| Line | Purpose |
| ---- | ------- |
| SDA  | Data    |
| SCL  | Clock   |

Both lines use **pull-up resistors**.

Why?

Because I2C uses **open-drain outputs**:

- Devices can pull line LOW
- No device actively drives HIGH

HIGH is created by resistor.

---

## Addressing System

Each slave has a:

- 7-bit or 10-bit address

Communication sequence:

1. Start condition
2. Address + R/W bit
3. ACK from slave
4. Data transfer
5. Stop condition

---

## Why I2C Is Efficient

With only 2 wires, you can connect:

- Many sensors
- EEPROM
- RTC
- LCD modules

Each identified by address.

---

## Arbitration (Advanced Concept)

If two masters transmit simultaneously:

- Bus arbitration decides winner
- Device that detects mismatch stops transmitting

This enables **multi-master systems**.

---

## Speed Modes

| Mode       | Speed   |
| ---------- | ------- |
| Standard   | 100 kHz |
| Fast       | 400 kHz |
| Fast+      | 1 MHz   |
| High Speed | 3.4 MHz |

Slower than SPI, but more scalable.

---

## Strengths

✔ Only 2 wires

✔ Addressable devices

✔ Scalable

✔ Built-in ACK mechanism

---

## Weaknesses

✖ Slower than SPI

✖ More protocol overhead

✖ Slightly more complex

---

# SPI vs I2C – Engineering Comparison

| Feature      | SPI              | I2C           |
| ------------ | ---------------- | ------------- |
| Wires        | 4+               | 2             |
| Speed        | Very high        | Moderate      |
| Duplex       | Full             | Half          |
| Addressing   | No               | Yes           |
| Multi-device | Separate CS pins | Address-based |
| Complexity   | Low hardware     | More protocol |

---

## When to Choose Each

Choose SPI when:

- Speed is critical
- Large data blocks (display, SD card)

Choose I2C when:

- Many small sensors
- Limited pins
- Clean wiring required

# Debugging Techniques (Structured Engineering Approach)

Debugging in embedded systems is harder because:

- No operating system
- No advanced console
- Hardware + software interaction

---

## Levels of Debugging

### 1. Hardware-Level Debugging

Check:

- Power supply voltage
- Ground connections
- Loose wires
- Short circuits
- Current limits

Use:

- Multimeter
- Oscilloscope (advanced)

---

### 2. Firmware-Level Debugging

Use:

- Serial printing
- LED indicators
- Step-by-step testing

Strategy:

1. Test each module independently
2. Validate inputs
3. Validate outputs
4. Integrate gradually

---

## Logical Debugging

Common logical errors:

- Wrong comparison operator
- Missing break
- Incorrect threshold
- Blocking delays

Use incremental verification.

---

## Professional Debug Strategy

1. Reproduce issue
2. Isolate subsystem
3. Reduce system complexity
4. Confirm hypothesis
5. Apply minimal fix

Never randomly change code.

---

# 5️⃣ Interrupts – Conceptual Deep Understanding

## What Is an Interrupt?

An interrupt is a **hardware signal that temporarily pauses the main program** to handle urgent tasks.

Instead of constantly checking:

```
if(buttonPressed)
```

Hardware tells CPU immediately.

---

## Polling vs Interrupt

| Polling               | Interrupt         |
| --------------------- | ----------------- |
| CPU constantly checks | Hardware triggers |
| Wastes CPU time       | Efficient         |
| Slower reaction       | Immediate         |

---

## How Interrupt Works Internally

1. Event occurs (button, timer overflow)
2. CPU:
   - Finishes current instruction
   - Saves program counter
3. Jumps to ISR
4. Executes ISR
5. Returns to main program

This is called **context switching**.

---

# 6️⃣ ISR – Interrupt Service Routine

ISR is a special function:

- Triggered automatically
- Must execute quickly

---

## Rules of ISR

Inside ISR:

- ❌ No delay()
- ❌ No Serial printing
- ❌ No heavy computation
- ❌ No dynamic memory

Why?

Because:

- Other interrupts are blocked
- System timing can break

---

## Best Practice

ISR should:

- Set a flag
- Exit immediately

Main loop handles heavy logic.

Example concept:

```arduino
volatile bool flag =false;

ISR_function() {
flag =true;
}
```

---

## Types of Interrupts

- External interrupt (button)
- Timer interrupt
- Serial interrupt
- Pin change interrupt

---

# 7️⃣ When to Use Interrupts

Use interrupts when:

✔ Real-time response required

✔ Precise timing needed

✔ Event-driven system

Avoid when:

✖ Task is slow

✖ System is simple

---

# 8️⃣ System-Level Integration Insight

When integrating:

- SPI devices may require timing control.
- I2C devices share bus → handle address conflicts.
- Interrupts may conflict with communication timing.
- Debugging must isolate communication layers.

A professional embedded system designer thinks about:

- Bus conflicts
- Latency
- Power integrity
- Interrupt priorities
- Resource management
